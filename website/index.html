Introduction

There is so much more to testing than just checking whether your code behaves as expected. Tests help you design your system. They specify the behavior of your classes. They give examples on how to use your API. They are a reliable documentation of your code. They are executables specifications guiding the design of your system.

Jnario has the goal to make writing executable specifications for Java as simple as possible. It provides two different languages, one for writing high-level acceptance specifications and one for writing low–level unit specifications. Both are really easy to integrate into existing projects as they compile to plain JUnit tests. Jnario is based on Xtext and Xtend. This means you have the expressiveness of Xtend together with the first class tooling support provided by Xtext, such as refactoring, code completion and debugging.

Features

  • Full interoparability with Java.
  • Generation of textual specifications.
  • Self-explanatory assertions.
  • Easy to integrate into existing Java projects.
  • Full Eclipse support with debugging.

Acceptance Specifications

Acceptance specifications in Jnario focus on the design and communication part of developing software. You can describe the features of your application in a business readable language similar to Cucumber, but with one major difference: you don't need to implement a separate parser for your steps. Instead, you can directly inline the necessary code to make your specification executable.

 Feature: Addition

  In order to avoid silly mistakes
  As a math idiot
  I want to be told the sum of two numbers
  
  Scenario: Add two numbers
    Calculator calculator = new Calculator()
    Given I have entered "50" into the calculator
      calculator.enter(args.first)
    And I have entered "70" into the calculator
    When I press "add"
      calculator.press(args.first)
    Then the result should be "120" 
      calculator.result => args.first

Unit Specifications

The goal of Jnario is to make it possible to write unit tests without boilerplate. Compare the following example specification to the same example implemented in Java.

package example

import java.util.Stack

describe Stack{
  context "empty stack"{
    fact subject.size should be 0
    fact subject.pop throws Exception
  }
  context "stack with one element"{ 
    before subject.add("element")
    fact subject.size should be 1
    fact "pop decreases size"{
      subject.pop
      subject.size => 0 
    }
  }   
}
